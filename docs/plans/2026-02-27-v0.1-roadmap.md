# CC-Manager v0.1 Roadmap — R25 to R35

**Goal**: Ship a production-quality open-source v0.1 that real users can install, configure, and use confidently.

**Standard**: Highest quality open-source project — on par with tools like Hono, Commander.js, tsx.

**Date**: 2026-02-27

---

## Real User Journey (Design North Star)

Every decision in this roadmap is validated against this user journey:

```
1. DISCOVER  → GitHub README answers "what is this?" in 10 seconds
2. INSTALL   → `npx cc-manager --repo .` works on first try
3. CONFIGURE → Sensible defaults, clear CLI help, no hidden requirements
4. SUBMIT    → Dashboard form or `curl` to submit first task
5. MONITOR   → Real-time dashboard shows progress, cost, status
6. DEBUG     → When tasks fail, error messages tell user WHY and WHAT TO DO
7. TRUST     → Cost controls, timeouts, graceful shutdown — user feels safe
8. SCALE     → Add workers, adjust budgets, batch submit — user grows usage
```

## Round-by-Round Plan

### R25 — Input Validation & Error Messages (Reliability)
**User story**: "When I submit bad data or something breaks, I get a clear error — not a 500 crash."

| # | File | Task | Why |
|---|------|------|-----|
| 1 | server.ts | Add Zod-style input validation for POST /api/tasks (prompt required, timeout > 0, maxBudget >= 0, priority enum) | Users will POST malformed JSON — validate early, return 400 with clear message |
| 2 | server.ts | Add global error handler middleware that catches unhandled exceptions and returns structured JSON error `{error, message, status}` instead of stack traces | Users must never see raw stack traces |
| 3 | scheduler.ts | Add getRunSummary() if not done in R24; validate task IDs exist before retry/cancel | Prevent cryptic errors when user references wrong task ID |
| 4 | store.ts | Add database migration system — version check on startup, auto-migrate schema | Users upgrading from older version must not lose data |
| 5 | index.ts | Validate --repo exists and is a git repo on startup. Print clear error: "Error: /path is not a git repository. Run from a git repo or use --repo <path>" | #1 user confusion point — running in non-git directory |

## Constraints

- **Budget**: ~$2-3 per round = ~$25-30 for R25-R35
- **Time**: 5 tasks per round, ~3 min per round execution
- **Quality**: Build must pass after every round
- **Risk**: Dashboard tasks have higher failure rate (HTML is tricky for agents)
- **Mitigation**: Dashboard rounds (R27-R28) get extra timeout (300s)

## Anti-Patterns to Avoid

1. **Feature creep** — v0.1 is not v1.0. Ship working basics, iterate.
2. **Over-engineering** — No plugin system, no multi-repo, no external reviewers yet.
3. **Cosmetic over substance** — Fix bugs before adding features.
4. **Untested features** — Every new method gets a corresponding test.
5. **Breaking changes** — Maintain API compatibility within v0.x.
